## 二、先讲清楚：四个9 是什么指标
可用性指标，行业通用：
- 3个9：99.9% → 每年允许宕机 525.6 分钟
- **4个9：99.99%** → 每年最多不可用 **52.56 分钟**
- 5个9：99.999% → 每年最多不可用 5.256 分钟

京东/阿里/拼多多核心交易、订单、物流、商品接口，**硬性要求：4个9起步，核心链路5个9**。

多级缓存不只是「快」，同时解决三件事：
1. **响应极快**：压垮时延（70ms 以内）
2. **扛流量**：挡住DB压力，防止数据库雪崩
3. **高可用兜底**：Redis挂了走本地缓存、DB挂了走缓存兜底，保障4个9可用性

## 三、为什么多级缓存能把接口压到 70ms 以内？
### 1. 你当前架构链路耗时对比
- 查 MySQL：**20～100ms+**，还要加锁、连接、磁盘IO
- 查 Redis（网络+内存）：**1～3ms**
- 查本地Caffeine缓存（进程内内存）：**0.01ms 级别，无网络**

### 2. 标准多级缓存查询流程（你现在写的就是大厂标准写法）
```
入参 →
1. 先查 本地Caffeine（一级缓存）
   → 命中：直接返回，微秒级
2. 本地未命中 → 查 Redis 二级缓存
   → 命中：返回，毫秒级
3. Redis未命中 → 查数据库（冷源）
   → 查到后：回填 Redis + 回填 本地缓存
   → 返回数据
```

**绝大多数热点运单、商品、用户数据，99% 命中在 本地/Redis**
根本打不到数据库，所以整体接口耗时稳稳：
**P99 < 50～70ms**，完全对标京东面试说的指标。

## 四、京东为什么死磕「多级缓存」？
1. **热点数据放大**
   订单、物流、商品都是典型**读多写少**，缓存收益极高
2. **削峰抗爆**
   大促、高峰流量，靠本地缓存直接拦截，不打Redis、不打DB
3. **异地/多机房容灾**
   Redis集群故障、短暂超时，本地缓存可以临时兜底，不雪崩
4. **时延刚性约束**
   移动端用户体感极强：
   > 100ms 以内无感，100～300ms 明显卡顿，500ms+ 流失用户
   所以大厂强制：**核心查询接口 P99 ≤ 70ms**
