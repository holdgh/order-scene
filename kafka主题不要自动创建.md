## 一、核心原则：生产环境 **永久禁止自动创建 Topic**
1. Kafka 服务端配置
```properties
auto.create.topics.enable=false
```
2. 理由（硬核、面试官必问）
- 自动创建 Topic 默认：**1分区 + 1副本**，无任何高可用、并发能力
- 分区数、副本数、消息保留时间、压缩策略、清理策略、最大消息尺寸，全部是默认垃圾配置
- 线上流量上来：消费瓶颈、分区不均、故障无法容灾、数据丢失风险极高
- 微服务多团队协作：乱建Topic、命名混乱、无人维护，运维灾难

---

## 二、业界标准正规做法（全环境统一规范）
### 1. 所有环境统一：**手动/脚本 预先创建 Topic**
开发环境、测试环境、预发、生产，**一律提前建好**
不依赖业务代码、不依赖Kafka自动创建。

### 2. 统一管控三件套
- 规范 Topic 命名（比如：`logistics-track-topic`）
- 提前规划：**分区数、副本数、过期时间、压缩、堆积策略**
- 权限管控：哪些业务组可读/可写

## 三、不同环境 标准实操方案

### ① 本地开发环境（你现在的场景）
目标：方便开发、不折腾、不乱配置
✅ 标准做法：
1. Kafka 本地 `server.properties`
```properties
auto.create.topics.enable=false
```
2. 本地启动后，**执行一次命令手动建Topic**
```bash
# 示例：轨迹 Topic 本地开发配置
kafka-topics.bat --create ^
--topic track-topic ^
--bootstrap-server 127.0.0.1:9092 ^
--partitions 3 ^
--replication-factor 1
```
- 分区给3～5，满足本地多线程消费测试
- 副本1（本地单节点，只能1）

3. 代码配置
```yaml
spring:
  kafka:
    listener:
      missing-topics-fatal: true
```
本地也强制校验，**养成好习惯**，避免上线踩坑。

---

### ② 测试/预发环境
✅ 标准做法：
- 运维/中间件团队统一脚本创建Topic
- 配置标准化：
    - 分区数：根据业务并发预估（10～20）
    - 副本数：**2（集群至少3节点）**
    - 消息保留：72h
    - 开启压缩、日志清理策略

---

### ③ 生产环境（严格规范）
✅ 标准做法：
1. 集群多节点，副本数统一 `3`
2. 分区数按**峰值并发、消费并行度**设计
3. 关闭服务端自动创建
4. 上线前**工单申请 + 运维统一创建Topic**
5. 绑定ACL权限：业务服务只能读写自己的Topic

## 四、那代码里如何优雅处理？（你项目里的写法）
### 方案A【推荐、最标准】
开发/生产全部提前建Topic，代码保持严格校验：
```yaml
spring.kafka.listener.missing-topics-fatal: true
```
好处：
- 启动即发现配置问题、Topic缺失，提前暴露
- 杜绝线上少建Topic导致服务启动雪崩

### 方案B【仅临时妥协，不推荐长期】
实在临时开发懒得建，临时关闭校验：
```yaml
spring.kafka.listener.missing-topics-fatal: false
```
⚠️ 禁止开 `admin.auto-create` + 服务端自动创建。

## 五、补充：什么时候用 `KafkaAdmin` 代码创建？
仅限**中间件平台、自研运维系统、内部管理平台**
业务普通微服务，**绝对不要在业务代码里写自动建Topic逻辑**。

业务服务职责：**生产/消费消息**
Topic 运维职责：**中间件&运维团队**

---

## 六、极简背诵版（面试直接说）
1. 生产环境关闭Kafka自动创建：`auto.create.topics.enable=false`
2. 所有环境Topic统一**提前手动/脚本预置**
3. 按业务并发规划分区数，集群环境副本数至少2~3，保障高可用
4. 业务代码不负责创建Topic，只负责收发消息
5. 代码开启`missing-topics-fatal=true`，提前校验环境完整性
